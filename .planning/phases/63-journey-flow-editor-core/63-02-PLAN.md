```typescript
interface JourneyFlowNodeDetailProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  node: JourneyFlowNode | null;
  isReadOnly?: boolean;
  onFieldChange?: (id: string, updates: Partial<JourneyFlowNode>) => void;
  onDeleteNode?: (id: string) => void;
}
```

Dialog content (`sm:max-w-[480px]`, return null when `node` is null):
1. **Name** — header `Input` exactly like the old dialog (borderless, text-lg font-semibold, placeholder 'Screen name'), `onChange → onFieldChange?.(node.id, { name: e.target.value })`. Include sr-only DialogTitle/DialogDescription for a11y.
2. **UI Type** — labeled row; `DropdownMenu` whose trigger is an outline Button showing `UI_TYPE_LABELS[node.uiType]` + chevron-down icon; `DropdownMenuItem` per entry of `UI_TYPE_LABELS` calling `onFieldChange?.(node.id, { uiType })`. (No shadcn Select exists in this project — DropdownMenu is the established picker.)
3. **Priority** — same DropdownMenu pattern over `PRIORITY_LABELS`, with a colored dot per option: must-have `var(--destructive)`, should-have `var(--primary)`, nice-to-have `var(--muted-foreground)` (inline style backgroundColor — do NOT copy the old dialog's `bg-blue-400` hardcode).
4. **Purpose** — labeled `Textarea` (rows 3, placeholder 'What does this screen do for the user?'), `onFieldChange?.(node.id, { purpose: e.target.value })`.
5. **Key elements** — labeled `Textarea` (rows 4, placeholder 'One element per line…') whose value is `node.keyElements.join('\n')` and whose onChange does `onFieldChange?.(node.id, { keyElements: e.target.value.split('\n') })`. NOTE: do not filter empty lines on every keystroke (that eats the newline the user just typed) — filter blanks on dialog close instead: in `onOpenChange`, when closing, call `onFieldChange?.(node.id, { keyElements: node.keyElements.map(s => s.trim()).filter(Boolean) })` before propagating.
6. **Addresses pain** — read-only display row (muted label + text), rendered only when `node.addressesPain` is set (this field is AI-populated in Phase 64, not user-edited).
7. **Footer** — destructive 'Delete screen' Button (trash icon) calling `onDeleteNode?.(node.id)` then `onOpenChange(false)`, plus an outline 'Close' button. When `isReadOnly`, render all inputs disabled and hide the delete button.

All field changes write through immediately (controlled by store state — the parent passes the live node), matching the old dialog's write-through pattern. No local form state, no save button.
