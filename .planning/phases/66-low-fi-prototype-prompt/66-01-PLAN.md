Define and export:

```typescript
import type { JourneyFlowNode, JourneyFlowEdge, JourneyFlowUiType, JourneyFlowPriority } from './types';
import { UI_TYPE_LABELS } from './types';

export interface ScreenSpec {
  name: string;
  uiType: JourneyFlowUiType;
  purpose: string;
  keyElements: string[];
  addressesPain?: string;
  priority: JourneyFlowPriority;
}

export interface NavLink {
  from: string; // source node name
  to: string;   // target node name
}

/** Parse a saved flow into screens + navigation. EXCLUDES annotation nodes (isAnnotation: true). */
export function parseScreensFromFlow(
  nodes: JourneyFlowNode[],
  edges: JourneyFlowEdge[]
): { screens: ScreenSpec[]; nav: NavLink[] }

/** Deterministic plain-text section: one block per screen (### name, UI type label, purpose, key elements, addressesPain, priority). */
export function buildScreenDescriptions(screens: ScreenSpec[]): string

/** Deterministic plain-text section: "- {from} → {to}" lines. Returns '' when nav is empty. */
export function buildNavigationSection(nav: NavLink[]): string
```

Implementation details:
- `parseScreensFromFlow`: filter `nodes.filter(n => !n.isAnnotation)` BEFORE mapping to ScreenSpec. For nav, build `nameById = new Map(nodes.map(n => [n.id, n.name]))` from ALL nodes, map edges to `{ from, to }`, then filter out links where either endpoint is empty OR either endpoint is an annotation node (build a Set of annotation node ids and exclude edges touching them).
- `buildScreenDescriptions`: for each screen emit `### {name}`, `Type: {UI_TYPE_LABELS[uiType]}`, `Purpose: {purpose}` (skip if empty), `Include: {keyElements.join(', ')}` (skip if empty array), `Addresses: {addressesPain}` (skip if undefined), blank line between screens.
- `buildNavigationSection`: heading-free body — just the link lines. Callers add their own heading.
- File-top doc comment: state this module is the PROMPT-04 shared boundary — fidelity-agnostic; low-fi and hi-fi prompt files compose these helpers with their own preambles/assemblers.

<task type="auto">
  <name>Task 2: Create low-fi prompt file (preamble + Gemini meta-prompt + fallback assembler)</name>
  <files>src/lib/ai/prompts/low-fi-prototype-prompt.ts</files>
  <action>
Create `src/lib/ai/prompts/low-fi-prototype-prompt.ts`. Imports shared helpers from `@/lib/journey-flow/prompt-builder` and types from `@/lib/journey-flow/types`. Do NOT import anything from `src/lib/ai/prompts/journey-v0-prompt.ts` or `src/lib/journey-mapper/*` — those are old-mapper types being parked in Phase 67.

Export four functions and one type:

1. `buildLowFiPreamble(): string` — the mandatory hard-rules wireframe preamble (locked decision: non-negotiable rules). Use exactly this content:

```
WIREFRAME PROTOTYPE INSTRUCTIONS (non-negotiable — do not deviate):
- Render in grayscale only. No color except black, white, and grays.
- Use the system default font. No custom fonts, icons, or branded assets.
- All UI elements are boxes and outlines only. No drop shadows, gradients, or polish.
- Placeholder images are labeled gray boxes (e.g. "[Image]", "[Logo]"). No real images.
- Include ONLY the elements needed to test the core idea. Strip everything decorative.
- Build a clickable prototype in your default stack: a single self-contained app, no backend, no external APIs.
- The goal is a clickable wireframe — sketch quality is correct and expected.
```

2. `buildLowFiGeminiPrompt(input: { brief: string; screens: ScreenSpec[]; nav: NavLink[]; testScope: TestScope; conceptName?: string | null }): string` — the meta-prompt SENT TO Gemini. It instructs Gemini to write the BODY of a prototype prompt for an AI coding agent, containing:
   - A "WHAT THIS PROTOTYPE TESTS:" section — 2-4 sentences covering who the user is, the problem, and the validation goal (what the wireframe is meant to prove), grounded ONLY in the supplied workshop brief.
   - A "SCREENS TO BUILD:" section — one block per screen, enriching each screen's purpose/keyElements into 2-4 concrete sentences describing what to render and what is clickable. Gemini must keep every screen, keep the exact screen names, never invent new screens, never drop screens.
   - A "NAVIGATION:" section reproducing the supplied links ("- From → To" lines) plus a one-line note on the user's path through the prototype.
   - When `testScope === 'feature'`: instruct Gemini to state the prototype covers a single feature mini-flow only (entry → action → result), not the full product. Include `conceptName` if provided.
   Embed the deterministic data via `buildScreenDescriptions(screens)` and `buildNavigationSection(nav)` plus the raw `brief`. Hard output rules at the end of the meta-prompt: plain text only, NO markdown code fences, NO styling/visual/branding instructions (a separate preamble handles style), NO tech-stack requirements, NO sample-data instructions.

3. `assembleLowFiPrompt(body: string, opts: { testScope: TestScope }): string` — composes the FINAL prompt text: `buildLowFiPreamble()` + blank line + (when testScope==='feature') the deterministic line `SCOPE: Single-feature prototype — build only the mini-flow screens below, not a full product journey.` + blank line + `body.trim()`. Prepending the preamble deterministically guarantees PROMPT-02's mandatory preamble even if Gemini ignores instructions.

4. `buildFallbackBody(input: { brief: string; screens: ScreenSpec[]; nav: NavLink[] }): string` — deterministic non-LLM body used when Gemini fails: "WHAT THIS PROTOTYPE TESTS:" + first ~8 lines of the brief, then "SCREENS TO BUILD:" + `buildScreenDescriptions(screens)`, then "NAVIGATION:" + `buildNavigationSection(nav)` (omit section when nav empty).

5. `export interface StoredPrototypePrompt { promptText: string; generatedFromFlowUpdatedAt: string; testScope: TestScope; selectedConceptId?: string; generatedAt: string; }` — the build_packs JSON content shape (timestamp staleness per research Pattern 4; Claude's-discretion choice: timestamps, zero schema changes).

  <name>Task 3: Create POST /api/build-pack/generate-prototype-prompt route</name>
  <files>src/app/api/build-pack/generate-prototype-prompt/route.ts</files>
  <action>
Create the API route following the established pattern in `src/app/api/build-pack/generate-journey-flow/route.ts` (auth → load → Gemini with fallback → upsert → respond). `export const maxDuration = 60;` at top (Pitfall 4).

Flow:
1. Parse body `{ workshopId?: string }`. 400 if missing.
2. Clerk `auth()`; 401 if no userId. Load workshop (`id, clerkUserId, title`); 403 if `clerkUserId !== userId` (owner-only — participants are read-only and never call this).
3. Load the Journey Flow row: `db.select().from(buildPacks).where(and(eq(buildPacks.workshopId, workshopId), like(buildPacks.title, 'Journey Flow:%')))`; find `formatType === 'json' && content`. If no row or `JSON.parse` fails or `parsed.nodes` empty → 400 `{ error: 'Journey Flow not found. Build and save your Journey Flow first.' }`.
4. Capture `journeyFlowUpdatedAt = journeyFlowRow.updatedAt.toISOString()` — this becomes `generatedFromFlowUpdatedAt`.
5. `const { screens, nav } = parseScreensFromFlow(parsed.nodes, parsed.edges ?? [])`. If `screens.length === 0` (only annotation nodes) → 400 with a clear error.
6. `const brief = await loadValidationBrief(workshopId)` from `@/lib/validation/llm-context`.
7. `const testScope: TestScope = parsed.testScope ?? 'journey'; const selectedConceptId = testScope === 'feature' ? parsed.selectedConceptId : undefined;`
8. Gemini path inside try/catch:
   - `const metaPrompt = buildLowFiGeminiPrompt({ brief: brief.brief, screens, nav, testScope, conceptName: selectedConceptId ?? brief.conceptName });`
   - `const result = await generateTextWithRetry({ model: google('gemini-2.5-flash-lite'), temperature: 0.4, prompt: metaPrompt });`
   - `recordUsageEvent({ workshopId, stepId: 'validate', operation: 'generate-prototype-prompt', model: 'gemini-2.5-flash-lite', inputTokens: result.usage?.inputTokens, outputTokens: result.usage?.outputTokens });`
   - Strip markdown fences from `result.text` (same regex chain as generate-journey-flow lines 182-186). If stripped text shorter than 100 chars, throw to fallback.
   - catch → `console.warn('[generate-prototype-prompt] LLM failed, using deterministic fallback:', ...)`; `body = buildFallbackBody({ brief: brief.brief, screens, nav })`.
9. `const promptText = assembleLowFiPrompt(body, { testScope });`
10. Build `StoredPrototypePrompt` content: `{ promptText, generatedFromFlowUpdatedAt: journeyFlowUpdatedAt, testScope, selectedConceptId, generatedAt: new Date().toISOString() }`. Upsert into build_packs under title `Prototype Prompt:${workshop.title || 'Product'}` with `formatType: 'json'` — exactly the generate-journey-flow upsert pattern (find existing json row with `like(buildPacks.title, 'Prototype Prompt:%')`, update content, else insert). NO cache check — this route ALWAYS generates fresh and overwrites (locked decision: regenerate overwrites, no version history; persistence-for-reload is the page's job).
11. Return 200 `{ promptText, buildPackId, generatedFromFlowUpdatedAt: journeyFlowUpdatedAt, testScope, selectedConceptId, usedLlm }`.
12. Outer try/catch → 500 `{ error: 'Failed to generate prototype prompt' }`.

Use plain `db` (fast-path write route convention per Phase 63 decision). Never import `create-v0-chat`/`deploy-v0` or anything from `src/lib/journey-mapper/`.
