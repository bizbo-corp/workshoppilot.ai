## Decisions Made

- **Fresh-gen only scope for Plan A:** The existing `loadFirstAssistantMessage` greeting-replay branch is being deleted by Plan B Task 3; inserting logs there would be wasted work. Plan A intentionally does not log replays.
- **Hoisted requestId binding:** `let requestId: string | null = null;` declared before any branching so Plan B's won-greeting fresh-gen path can assign to the same binding and reuse the onFinish backfill block without duplication.
- **Observability must not break chat:** All DB writes wrapped in `try/catch` with `console.error` — a logging failure never surfaces to the user or aborts the stream.
- **TTL deferred:** No `vercel.json` cron or `src/app/api/cron/` infra exists in repo. At current usage rates, data accumulation is benign for weeks-to-months. Flagged as follow-up.
- **Admin join by scope+timestamp:** The join does not depend on `response_message_id`, so the AI SDK v5 null-id limitation (see below) does not affect the admin UI.

## Known Limitations

### AI SDK v5: response_message_id reliably stays null

**What happens:** In `result.toUIMessageStreamResponse({ onFinish })`, the `messages` argument received by `onFinish` is the server-side view of the conversation. AI SDK v5 generates the new assistant message ID **client-side** — the server's `onFinish` sees `responseMessages[last].id === ''` (empty string). The backfill UPDATE therefore sets `response_message_id` to `null` (the guard `if (typeof requestId === 'string' && assistantMessageId)` fails on empty string).

**Evidence:** Confirmed in dev: all `crl_*` rows show `response_message_id IS NULL` despite successful streaming.

**What is unaffected:** The admin UI, the scope-join query, and all other captured columns (`system_prompt`, `system_prompt_sha`, `model_messages_json`, scope identifiers) are fully correct. The observability goal (inspecting the rendered prompt for any feedback row) is fully met.

**Plan B inheritance:** Plan B's won-greeting fresh-gen path uses the same hoisted `requestId` + `onFinish` backfill, so it inherits the same null behavior. The race-loser replay path inserts with `filled.messageId` directly (from the cached message) and is unaffected.

**Recommended fix paths (future follow-up, not in this plan):**
1. Route backfill through `autoSaveMessages` (the client-side hook that receives the final assistant message with its client-assigned id). That hook already knows the client id and could POST to `/api/admin/chat-request-log` to update the row.
2. Investigate `streamText`'s `messageId` override option in AI SDK v5 — if the server can force the assistant message id before streaming starts, both server and client would share the same id.
3. Accept null and use `request_id` as the primary correlation key (it is already indexed and unique per request).

**In-code annotation:** `src/app/api/chat/route.ts` lines 254-258 contain a comment pointing readers to this SUMMARY and describing the limitation, committed as `d60f090`.
