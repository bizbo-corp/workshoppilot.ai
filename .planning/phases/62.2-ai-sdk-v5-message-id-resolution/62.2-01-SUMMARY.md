# Phase 62.2 Plan 01: AI SDK v6 generateMessageId — Server-Side Message-ID Resolution Summary

**AI SDK v6 generateMessageId wired through toUIMessageStreamResponse, eliminating three 62.1 workarounds: empty-id filter in saveMessages, post-stream backfill UPDATE block, and deterministic placeholder id carried as canonical assistant message id.**

## Performance

- **Duration:** ~2 sessions (Tasks 1-2 in first agent, Task 3 human-verified by orchestrator)
- **Started:** 2026-05-16
- **Completed:** 2026-05-16
- **Tasks:** 3 (Tasks 1-2 automated; Task 3 manual verification — approved)
- **Files modified:** 2

## Accomplishments

- Pre-generate `responseMessageId = createPrefixedId('msg')` at fresh-gen entry in `/api/chat` POST handler; pass as `generateMessageId: () => responseMessageId!` to `toUIMessageStreamResponse` so the AI SDK v6 wire-format `start` chunk carries the server-chosen id to the client — `useChat` adopts it via `state.message.id = chunk.messageId`
- `chat_request_logs.response_message_id` populated at insert time (no `null` placeholder, no post-stream backfill UPDATE block) — AC-2 satisfied at the point of insert
- Empty-messageId filter `.filter((row) => row.messageId.length > 0)` removed from `saveMessages`; `chat_messages_message_id_nonempty_chk` CHECK constraint now acts as a loud canary if misconfiguration ever lets an empty id slip through
- `fillGreetingPlaceholder` now called with `assistantMsg!.id` (real AI SDK-generated id) instead of `greetingClaim.messageId` — the placeholder row's `messageId` column is replaced with the canonical AI SDK id on stream completion; the client's `autoSaveMessages` subsequent write hits the same id and is silently dropped by `onConflictDoNothing()`, eliminating the dual-row state from 62.1 Known Limitation 4

## Decisions Made

- Use `createPrefixedId('msg')` (project helper from `@/lib/ids`) not `createIdGenerator` — consistent with `crl_…` prefix convention already in place
- `originalMessages: messages` kept alongside `generateMessageId` — without it AI SDK v6 short-circuits id injection (source: `node_modules/ai/dist/index.mjs:4751-4761`); this was documented as Pitfall #1 in 62.2-RESEARCH.md and the verify check explicitly asserts it is present
- `fillGreetingPlaceholder` argument changed from `greetingClaim.messageId` to `assistantMsg!.id` — the only caller-side change; function signature unchanged
