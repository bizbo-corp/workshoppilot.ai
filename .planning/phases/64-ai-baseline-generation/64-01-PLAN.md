2. `detectTwoSided(challengeContext: string, concepts: ConceptData[]): { isTwoSided: boolean; riskierSide: string }` — scan lowercased combined text for distinct keyword signals: `marketplace`, `platform`, `two-sided`, `host`, `guest`, `provider`, `consumer`, `buyer`, `seller`, `supply`, `demand`, `b2b2c`. Require >= 2 DISTINCT keyword hits for `isTwoSided: true`. `riskierSide`: default `'supply/provider side'` (the side that must be convinced to participate) unless demand-side keywords dominate, then `'demand/consumer side'`. Return `{ isTwoSided: false, riskierSide: '' }` otherwise.

3. `detectArchetype(challengeContext: string, concepts: ConceptData[], personaRole?: string | null): FlowArchetype` — call `detectStrategicIntent(...)` + `normalizeIntent(...)`, then refine with archetype keyword signals over the combined text (per 64-RESEARCH "Archetype Detection from Workshop Signals" table):
   - loop keywords (`habit`, `log`, `track`, `streak`, `daily`, `retention`, `come back`, `feedback loop`) → `'loop'`
   - branching keywords (`eligibility`, `triage`, `approve`, `reject`, `qualify`, `choose your`, `decision`) → `'branching'`
   - funnel keywords (`landing page`, `sign-up`, `signup`, `waitlist`, `fake door`, `desirability`, `conversion`) AND intent is marketing → `'funnel'`
   - intent `'marketing-site'` (no funnel signal) → `'single-page-sections'`
   - intent `'tool'` → `'single-screen-tool'`
   - intent `'dashboard'` or `'admin-portal'` → `'hub-and-spoke'`
   - otherwise (`'web-app'` etc.) → `'linear-sequence'`
   Check loop/branching/funnel keyword overrides BEFORE the intent defaults (they are more specific signals).

4. `normalizeUiType(value: unknown): JourneyFlowUiType` — if value is one of the 12 valid `JourneyFlowUiType` strings return it, else `'detail-view'`.

5. `layoutPositions(archetype: FlowArchetype, count: number): Array<{x: number; y: number}>` — deterministic per-archetype placement (research Pattern 7):
   - `linear-sequence` / `funnel` / `branching` backbone: row, `x = i * 340, y = 160`
   - `hub-and-spoke`: hub at `(400, 160)`, spokes on a circle radius 320 at evenly spaced angles
   - `single-page-sections`: column, `x = 400, y = i * 220`
   - `single-screen-tool`: `(200, 160)` and `(600, 160)`
   - `loop`: N points on a circle radius `max(220, N * 60)` centered `(400, 300)`

6. `buildAnnotationNode(riskierSide: string): JourneyFlowNode` — id `'jf-node-annotation-two-sided'`, name `'Two-Sided Product Detected'`, `uiType: 'modal'`, `isAnnotation: true`, `priority: 'must-have'`, `keyElements: []`, position `{ x: 400, y: -120 }`, purpose explaining: this looks like a two-sided product; this baseline maps only the {riskierSide} — the riskier side to test first; the other side is intentionally out of scope.

7. The main entry:
```typescript
export interface FlowGenerationInput {
  concepts: ConceptData[];
  challengeContext: string;
  persona?: { name?: string; pains?: string[] } | null;
  scope: TestScope;
  selectedConceptName?: string;  // feature mode: which concept
}
export interface FlowGenerationOutput {
  nodes: JourneyFlowNode[];      // includes annotation node when two-sided
  edges: JourneyFlowEdge[];
  flowArchetype: FlowArchetype;
  strategicIntent: StrategicIntent;   // ARCHETYPE_TO_INTENT[archetype], except admin-portal override when intent-detection said admin-portal under hub-and-spoke
  isTwoSided: boolean;
}
export function heuristicGenerateFlow(input: FlowGenerationInput): FlowGenerationOutput;
```

Node names/purposes derived from the selected concept (match by `selectedConceptName` against concept `name`, fall back to first concept).
   - **Journey mode:** `detectArchetype`, then generate per archetype: linear → 4-6 screens (entry/auth-or-onboarding/core screens from concepts/result); hub-and-spoke → 1 hub ('dashboard') + one spoke per concept feature (cap 5) with hub→spoke edges; single-page-sections → 4-5 section nodes connected top-to-bottom; funnel → exactly 3-4 nodes (landing with strong-CTA purpose → sign-up form → confirmation/thank-you); branching → linear backbone + 1 fork (two targets from one decision node); single-screen-tool → 2 nodes (input → result); loop → 3-5 nodes with a closing edge from last back to first.
   - Node ids: `jf-node-gen-{i}`; edge ids: `jf-edge-gen-{i}`. Positions from `layoutPositions`. Every node gets non-empty `name`, `purpose`, valid `uiType`, `priority` ('must-have' for first 3, 'should-have' after), `keyElements` (2-4 short strings derived from concept features/usp where available, else sensible defaults for the screen type).
   - Two-sided: run `detectTwoSided`; when true, append `buildAnnotationNode(riskierSide)` (no edges to it) and keep generating only one side.
