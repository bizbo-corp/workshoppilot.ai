### 🚧 v1.9 Multiplayer Collaboration (In Progress)

**Milestone Goal:** Add real-time multiplayer workshops where a human facilitator leads 5-15 participants through design thinking on a shared live canvas with presence, cursors, and facilitator controls.

**Phase Numbering:** Phases 54-58 (continuing from v1.8's Phase 53)

**Coverage note:** SYNC-04 (per-participant Crazy 8s slots) deferred to v2. Requires per-participant partitioned storage regions in Liveblocks — a fundamentally different data model from the shared-canvas architecture built in Phase 55. All remaining 20 v1.9 requirements are mapped.

- [x] **Phase 54: Liveblocks Foundation** - Install Liveblocks, configure types and room convention, add DB schema for multiplayer, register StorageUpdated webhook, establish dual-store factory pattern and lazy-loading wrapper (completed 2026-02-26)
- [x] **Phase 55: Core Canvas Sync** - Create multiplayer workshop type, wire Liveblocks CRDT storage into canvas store, sync all canvas nodes and edges in real time across participants, enforce EzyDraw single-editor lock (completed 2026-02-26)
- [x] **Phase 56: Live Presence** - Broadcast and render live cursors in flow coordinates, build participant list panel with online/idle status, surface join/leave toasts and facilitator badge (completed 2026-02-26)
- [ ] **Phase 57: Guest Auth and Join Flow** - Generate share links, build `/join/[token]` page with name-entry gate, implement guest HttpOnly cookie auth, add lobby waiting screen and state recovery on reconnect
- [ ] **Phase 58: Facilitator Controls** - Gate step progression and AI chat input to facilitator, implement viewport broadcast, countdown timer, and session end flow with final canvas persistence

## Phase Details

### Phase 54: Liveblocks Foundation
**Goal**: The real-time infrastructure is installed, typed, and wired into the build — every subsequent multiplayer feature has a solid base to build on without retrofitting
**Depends on**: Phase 53
**Requirements**: INFR-01, INFR-02, INFR-05
**Success Criteria** (what must be TRUE):
  1. `@liveblocks/client`, `@liveblocks/react`, and `@liveblocks/node` (all v3.14.0) are installed with no bundle size regression on solo workshops (confirmed via bundle analyzer)
  2. `src/lib/liveblocks/config.ts` exports typed Presence and Storage definitions plus a room-naming convention (`workshop-[id]`)
  3. `/api/liveblocks-auth` route exists and issues access tokens for authenticated Clerk users (guest path stubbed)
  4. `/api/webhooks/liveblocks` StorageUpdated handler is registered and writes canvas state to Neon `stepArtifacts` (verified in Liveblocks dashboard)
  5. Neon schema includes `workshopType` column on workshops, `workshop_sessions` and `session_participants` tables, and multiplayer components are behind `next/dynamic` with `ssr: false`
**Plans**: 3 plans

Plans:
- [ ] 54-01-PLAN.md — Install Liveblocks packages (v3.14.0), create typed config with global Presence/Storage/UserMeta augmentation, add env var validation
- [ ] 54-02-PLAN.md — Add workshopType column to workshops, create workshop_sessions and session_participants tables, run Drizzle migration
- [ ] 54-03-PLAN.md — Implement /api/liveblocks-auth (Clerk path), /api/webhooks/liveblocks StorageUpdated handler, and next/dynamic lazy-loading wrapper

### Phase 55: Core Canvas Sync
**Goal**: Two participants in the same multiplayer workshop see each other's canvas edits in real time — nodes created, moved, and deleted by one immediately appear on the other's screen
**Depends on**: Phase 54
**Requirements**: SESS-01, SYNC-01, SYNC-02, SYNC-03, SYNC-05
**Success Criteria** (what must be TRUE):
  1. Facilitator can create a multiplayer workshop from the dashboard (distinct option from solo AI-led mode)
  2. A sticky note created in one browser tab appears on the canvas in a second browser tab within 500ms
  3. A sticky note moved by one participant moves on all other participants' canvases in real time
  4. New post-it nodes created by a participant automatically inherit that participant's assigned color
  5. Opening EzyDraw on a shared drawing node locks that node — a second participant attempting to open it sees a "being edited" indicator and cannot open EzyDraw simultaneously
**Plans**: 3 plans

Plans:
- [x] 55-01-PLAN.md — Install @liveblocks/zustand + upgrade zustand to v5, create createMultiplayerCanvasStore factory, add multiplayer workshop creation flow to dashboard dialog, and multiplayer badge on cards
- [x] 55-02-PLAN.md — Wire CanvasStoreProvider to branch on workshopType with enterRoom/leaveRoom, disable auto-save in multiplayer mode, implement post-it color inheritance, EzyDraw single-editor lock via Presence, and webhook Drizzle upsert
- [ ] 55-03-PLAN.md — Gap closure: Add on-canvas visual "Being edited by [name]" overlay to locked drawing nodes (SYNC-05 indicator)

### Phase 56: Live Presence
**Goal**: Every participant can see who else is in the workshop and where their cursors are on the canvas in real time
**Depends on**: Phase 55
**Requirements**: PRES-01, PRES-02, PRES-03, PRES-04, PRES-05, PRES-06
**Success Criteria** (what must be TRUE):
  1. Moving the mouse over the canvas broadcasts the cursor to all other participants within 50ms (throttled, never raw mousemove)
  2. Each cursor displays the participant's name and their assigned color; the facilitator's cursor shows a distinct badge or icon
  3. A participant list panel shows all connected users with avatar initials, name, assigned color, and online/idle dot indicator
  4. A toast notification appears (bottom corner, non-blocking) when any participant joins or leaves the workshop
  5. Participants who have not interacted with the canvas for more than 2 minutes show a visible idle indicator in the participant list
**Plans**: TBD

Plans:
- [ ] 56-01: Add Presence type (cursor position in flow coordinates, name, color, role) to `liveblocks/config.ts`; implement `onMouseMove`/`onMouseLeave` handlers with 50ms throttle and `screenToFlowPosition` conversion; render `LiveCursors` component inside ReactFlow `ViewportPortal`
- [ ] 56-02: Build `PresenceBar` with `useOthers()` — avatar initials, name, online/idle dot, idle timer (>2 min), facilitator badge; wire join/leave toast notifications via Liveblocks event listeners

### Phase 57: Guest Auth and Join Flow
**Goal**: A participant can click a share link, enter their name, and land in a lobby — no Clerk account, no friction
**Depends on**: Phase 54
**Requirements**: SESS-02, SESS-03, SESS-04, INFR-03, INFR-04
**Success Criteria** (what must be TRUE):
  1. A facilitator can copy a share link from the multiplayer workshop (unique token, sufficient entropy) with one click
  2. Opening the share link in an incognito browser shows a name-entry modal before any workshop content is visible
  3. After entering a name, the participant lands on a lobby screen ("Waiting for the facilitator to start") — no workshop canvas visible until the facilitator begins
  4. A participant who loses network connection and reconnects sees the current canvas state fully restored without manual refresh
  5. Guest identity persists within the same browser tab (sessionStorage) — page refresh returns to the same lobby or canvas position without re-entering name
**Plans**: 2 plans

Plans:
- [ ] 57-01-PLAN.md — Guest cookie utility (HMAC-SHA256), /api/guest-join endpoint, /join/[token] page with GuestJoinModal (full-screen name entry), ShareButton in workshop header, Clerk middleware public route updates
- [ ] 57-02-PLAN.md — Guest path in /api/liveblocks-auth, /api/session-status/[token] polling endpoint, GuestLobby with auto-transition on session start, ReconnectionListener (toast-based disconnect/reconnect UX), workshop route middleware for guest canvas access

### Phase 58: Facilitator Controls
**Goal**: The facilitator has full control of the session — step progression, AI chat, viewport alignment, timers, and graceful session end — while participants cannot interfere with facilitation decisions
**Depends on**: Phase 56, Phase 57
**Requirements**: FACL-01, FACL-02, FACL-03, FACL-04, FACL-05, SESS-05
**Success Criteria** (what must be TRUE):
  1. Only the facilitator can advance or go back between steps — Next/Previous Step buttons are hidden for participants and the server action rejects calls from non-facilitators
  2. Only the facilitator sees and can use the AI chat input — participants see the conversation history read-only
  3. Facilitator clicking "Bring everyone to me" broadcasts current viewport position and zoom — all participant views animate to match within 500ms
  4. Facilitator can set a countdown timer (e.g., "5 minutes") visible to all participants with a shared countdown display
  5. When the facilitator ends the session, all participants immediately see a "Session has ended" full-screen overlay; final canvas state is persisted to Neon via Liveblocks REST API snapshot before the room is archived
**Plans**: TBD

Plans:
- [ ] 58-01: Implement `StepProgressionControl` (role-gated to facilitator via `useSelf().info.role`); update `advanceWorkshopStep` server action with server-side Clerk auth check against `workshops.clerkUserId`; implement facilitator-only AI chat input (participants see read-only view); broadcast `STEP_CHANGED` events via `useBroadcastEvent` with parallel Neon writes for late-joiner recovery
- [ ] 58-02: Implement "Bring everyone to me" viewport broadcast (`VIEWPORT_SYNC` event with x/y/zoom); implement countdown timer (facilitator-set, `useBroadcastEvent`, rendered for all participants); implement "End Session" with confirmation modal, Liveblocks REST API manual snapshot to Neon, `SESSION_ENDED` broadcast, and full-screen overlay on participant clients
