# Phase 62.2: AI SDK v5 Message-ID Server-Side Resolution

## Goal

Eliminate the three workarounds in `chat_messages` and `chat_request_logs` persistence that were introduced or exposed by phase 62.1. All three share a single root cause: AI SDK v5 generates the assistant message id **client-side**, so the server's `onFinish` callback sees `msg.id === ''` for newly streamed assistants.

## The three workarounds to remove

1. **Greeting placeholder messageId is permanent** (`src/lib/ai/message-persistence.ts` + `src/app/api/chat/route.ts` ~line 320).
   - `fillGreetingPlaceholder` keeps the deterministic `greeting:{session}:{step}:{participantId|fac}` messageId because it has no real assistant id to substitute.
   - Side effect: client's `useAutoSave` writes a parallel row with the AI-generated id — the same response stored twice with different message_ids.

2. **`saveMessages` drops server-side rows** (`src/lib/ai/message-persistence.ts` ~line 60).
   - Filters out rows where `messageId === ''` to satisfy `chat_messages_message_id_nonempty_chk` (the CHECK constraint added in migration 0024).
   - Without the filter, `onFinish` throws → stream returns 500 → `ERR_INCOMPLETE_CHUNKED_ENCODING`.
   - Reliance: client's `useAutoSave` server action writes the canonical row a few seconds later. Brief window of "server thinks the message doesn't exist."

3. **`chat_request_logs.response_message_id` is reliably null on fresh-gen paths** (`src/app/api/chat/route.ts` ~line 333).
   - onFinish's backfill is guarded by `if (typeof requestId === 'string' && assistantMessageId)`. Since `assistantMessageId === ''`, the UPDATE never runs.
   - Race-loser replay path is unaffected (uses `filled.messageId` directly).

## Acceptance criteria

1. **Zero empty-messageId rows** inserted into `chat_messages` from any server-side path. (Verify by removing the empty-id filter in `saveMessages` and confirming no CHECK violations occur in dev traffic over a test session.)
2. **`chat_request_logs.response_message_id` populated** for all fresh-gen requests. Currently 100% null in dev DB; should be 100% non-null after.
3. **`chat_request_logs.response_message_id` populated** for the replay path (already works — must continue working).
4. **Greeting placeholder pattern simplified** — if Option A is chosen, `fillGreetingPlaceholder` can stop using the deterministic placeholder messageId; the placeholder row becomes the canonical row with the AI-generated id.
5. **"Known Limitations" sections deleted** from `.planning/phases/62.1-.../62.1-01-SUMMARY.md` and `62.1-02-SUMMARY.md` (or updated to reflect closure).
6. **The CHECK constraint stays**. It's a DB-level invariant that this phase must satisfy without filtering.

## Suggested approaches (research phase evaluates both)

### Option A — Server-side message id generation (preferred if AI SDK v5 supports it)

Use `streamText`'s message-id override (likely `experimental_generateMessageId` or v5 equivalent) to generate the assistant message id server-side BEFORE streaming. The generated id flows through:
- `chat_request_logs` insert (already has hoisted `requestId`; this adds `responseMessageId` immediately, no backfill needed)
- `onFinish`'s `responseMessages[last].id` now has the real id
- `fillGreetingPlaceholder` uses the real id; placeholder row becomes the canonical row
- `saveMessages` filter becomes unnecessary
- Client's `useChat` accepts the server-generated id (verify this is the AI SDK v5 contract)

Most invasive but fixes the root cause cleanly.

### Option B — Route backfill through `autoSaveMessages`

Keep AI SDK v5 client-side id generation. Refactor so the client's `autoSaveMessages` server action handles the final write AND backfills `chat_request_logs.response_message_id` keyed on a scope+timestamp join (the same join the admin UI uses).

Less invasive but doesn't eliminate the dual-row duplication for greetings. The CHECK-filter in saveMessages stays. Still better than today because at least `chat_request_logs.response_message_id` becomes populated.

## Files likely touched

- `src/app/api/chat/route.ts` — remove the assistantMessageId guard, possibly pass `experimental_generateMessageId`
- `src/lib/ai/message-persistence.ts` — `fillGreetingPlaceholder` may simplify; `saveMessages` empty-id filter may be removable
- `src/lib/ai/chat-config.ts` — if a wrapper around `streamText` exists, propagate the id-generation option
- `src/actions/auto-save-actions.ts` — if Option B is chosen, accept a `requestId` parameter and backfill
- Possibly the `chat_request_logs` schema if Option B needs a new column (unlikely — scope+timestamp join suffices)

## Out of scope

- Removing the `chat_messages_message_id_nonempty_chk` CHECK constraint
- Removing the greeting unique index used by `claimGreetingPlaceholder`
- Changing the `__step_start__` trigger literal or the deterministic user-side trigger id (`step-start:{session}:{step}:{participantId|fac}`) — these remain stable per 62.1's non-negotiables

## Verification path

Once implemented, run the 5x-rapid-mount test from phase 62.1 (which was deferred). After this phase ships, exactly 1 chat_messages row should appear per scope (no client/server dual-row), and the chat_request_logs row should have a non-null response_message_id matching that single chat_messages row.
