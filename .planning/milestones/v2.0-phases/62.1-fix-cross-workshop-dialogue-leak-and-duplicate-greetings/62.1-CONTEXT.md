<domain>
## Phase Boundary

Hotfix patch under v2.0. Three bugs in `/api/chat` for the AI facilitation flow, surfaced when a user (admin) reported the stakeholder-mapping welcome message at workshop start was wrong:

1. **Duplicate AI greetings** — entering Step 2 of TrustStack workshop `ws_gorg6yevjenjk4zcyosdx2ye` / session `ses_u9yw80ayadx3sjnj7dy3tubh` persisted three assistant greeting rows in 2.4 seconds (two near-identical "professional storytelling" welcomes plus a third "storytelling to help professionals" welcome) plus a row with `message_id=""` (empty string).
2. **Cross-workshop content leak** — two of those welcomes contained a stale Step 1 challenge that has since been overwritten ("How might we leverage AI to enhance trust and transparency within professional storytelling"); the third contained the verbatim challenge from a totally different workshop (`ws_tapiuu5gx638xf5pa1v6brnt` "Story Teller Framework").
3. **Hallucination license in stakeholder-mapping prompt** — the prompt explicitly says "If either input is unclear in the summary, recover the closest version from the surrounding text," which is an open invitation to fabricate when context is missing.

In scope: `/api/chat` route, message-persistence helpers, stakeholder-mapping system prompt, chat-panel + participant-chat-panel client transports, step-summaries schema, dialogue-feedback admin UI, one new table (`chat_request_logs`), and migrations for the above.

Explicitly out of scope: any other AI step prompts beyond stakeholder-mapping (Bug B1's prompt fix is targeted because that's where the bug surfaced); replatforming useChat or transport; workshop/session schema changes; auth flow changes.

</domain>

## Implementation Decisions (locked by user)

### Roadmap slotting
- Insert as Phase 62.1 hotfix under v2.0 (decimal pattern, precedent: Phase 13.1). v2.0 stays "shipped"; this is a follow-on patch entry.

### Five workstreams, sequenced
The user pre-agreed the structure. Recommended sequence: 1 + 3 first (observability + most likely root cause), then 2 (kills duplicate symptom), then 5 (cheap), then 4 (forensic depth). Planner may adjust waves but workstreams are locked.

1. **Observability — request log table.** New `chat_request_logs` table with columns `(id, session_id, step_id, participant_id, workshop_id, request_id, system_prompt_sha, system_prompt, model_messages_json, response_message_id, created_at)`. Write inside `/api/chat` *before* `streamTextWithRetry`. Wire the admin dialogue feedback UI (`src/app/admin/dialogue/dialogue-feedback-client.tsx`) to display the captured prompt for each feedback row by joining on the most-recent matching `chat_request_logs` row (matching scope = `session_id + step_id + participant_id`, picking the row immediately preceding the feedback `created_at`). Add 14-day TTL cleanup. Files: new migration, new `src/db/schema/chat-request-logs.ts`, `src/app/api/chat/route.ts`, `src/app/admin/dialogue/dialogue-feedback-client.tsx`, possibly an admin API endpoint to fetch the log.

2. **Greeting singleton via DB lock.** Replace the read-then-act `loadFirstAssistantMessage` check with an atomic `INSERT ... ON CONFLICT DO NOTHING RETURNING id` against the existing unique index `chat_messages_session_step_participant_message_uniq`, using a deterministic placeholder row id `greeting:{session}:{step}:{participant}`. If insert returns 0 rows, this request lost the race — poll briefly (~3s, 100ms ticks) for the placeholder to be filled (content non-empty), then replay via the existing `createUIMessageStream` path. The winning request UPDATEs the placeholder row with the streamed content rather than inserting a new row. Add `CHECK (length(message_id) > 0)` constraint after a one-shot cleanup of existing empty-id rows. Files: `src/app/api/chat/route.ts`, `src/lib/ai/message-persistence.ts`, new migration, `src/components/workshop/chat-panel.tsx`, `src/components/workshop/participant-chat-panel.tsx`.

3. **Stop stale fetches across navigation/scope changes.** Three pieces:
   - (a) Abort in-flight `useChat` requests when `sessionId` or `workshopId` changes — use `AbortController` in the `transport` memo and call `useChat`'s `stop()` on cleanup. (Researcher must verify the exact AI SDK API surface.)
   - (b) Add `key={`${sessionId}:${workshopId}:${step.id}`}` on `<ChatPanel>` and `<ParticipantChatPanel>` in `src/components/workshop/step-container.tsx` so React forcibly remounts on scope change.
   - (c) Server-side assert in `/api/chat`: load `sessions.workshop_id` for the requested `sessionId` and assert it equals the body's `workshopId`. If not, return 409 with structured error `{ error: 'workshop_session_mismatch', sessionId, requestedWorkshopId, actualWorkshopId }` and write nothing to chat_messages.
   Files: `src/components/workshop/chat-panel.tsx`, `src/components/workshop/participant-chat-panel.tsx`, `src/components/workshop/step-container.tsx`, `src/app/api/chat/route.ts`.

4. **Append-only step summaries.** Migration: drop `unique(workshop_step_id)` on `step_summaries`, add `is_current boolean default true` and `superseded_at timestamptz`. New summary writes as new row + flips prior to `is_current=false, superseded_at=now()`. `assembleStepContext` reads `where is_current = true`. Find every writer with grep on `stepSummaries` insert/upsert sites. Files: new migration, `src/db/schema/step-summaries.ts`, `src/lib/context/assemble-context.ts`, all step-summary writers.

5. **Stakeholder-mapping prompt hardening.** Replace the "recover the closest version from the surrounding text" license in `src/lib/ai/prompts/steps/02_stakeholder_mapping.ts:80-85` with a hard refusal: if no Step 1 challenge summary, reply with a one-line "go back and confirm Step 1" message and emit no canvas tags / no suggestions. Plus: in `src/lib/context/assemble-context.ts`, when `summaries === ''` for a step that depends on prior summaries, inject a sentinel `"⚠️ NO PRIOR STEP CONTEXT AVAILABLE — DO NOT INVENT ONE"` string and `console.warn`. Files: `src/lib/ai/prompts/steps/02_stakeholder_mapping.ts`, `src/lib/context/assemble-context.ts`.

### Acceptance criteria for the phase
1. Every `/api/chat` request writes a `chat_request_logs` row; admin dialogue UI shows literal system prompt for any feedback row.
2. Mounting any step page 5× rapidly (or under React strict mode) → exactly 1 assistant greeting row in `chat_messages` AND exactly 1 Gemini call in `ai_usage_events`.
3. POSTing `/api/chat` with mismatched `sessionId`/`workshopId` → 409, no DB write.
4. Two browser tabs on different workshops cannot cross-contaminate chat history (verified via synthetic test).
5. Deleting a Step 1 summary row and entering Step 2 → "go back and confirm Step 1" reply, no canvas tags, no hallucination.
6. (If workstream 4 is included this phase) Editing a Step 1 summary preserves prior as `is_current=false`; queryable historical summaries.

### Constraints / non-negotiables
- Next.js 16 App Router, Drizzle ORM, Neon Postgres serverless driver (edge-compatible). Migrations via drizzle-kit (look at `drizzle/` for existing patterns).
- No breaking changes to existing chat history rows; migrations must be backwards compatible.
- The unique index `chat_messages_session_step_participant_message_uniq` already exists — leverage it, don't replace it.
- React 19 + AI SDK `useChat` from `@ai-sdk/react` with `DefaultChatTransport` from `ai`.
- The `__step_start__` literal trigger string and the deterministic user-side id `step-start:{session}:{step}:fac|<participantId>` must remain stable — they are referenced in multiple files and existing chat history.
- Empty-id cleanup migration must be idempotent and safe to re-run.
