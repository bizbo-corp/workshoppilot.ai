## Decisions Made

- **Skip abortSignal for greeting requests:** The placeholder row is inserted before streaming begins. If the client aborts mid-stream (Strict Mode remount, rapid navigation), `onFinish` would never fire and the placeholder row would remain empty — blocking future claims and forcing a 3s poll → fresh-gen recovery on every subsequent mount. Skipping `abortSignal` for `__step_start__` requests ensures the stream always completes server-side. `abortSignal: req.signal` still applies for user-typed messages, preserving token savings on user-initiated aborts.
- **Use placeholder's deterministic id in fillGreetingPlaceholder:** AI SDK v5 generates the assistant message id client-side; `onFinish` server-side receives `assistantMsg.id === ''`. Passing `greetingClaim.messageId` (the deterministic `greeting:{sessionId}:{stepId}:{participantId|fac}` id) fills the placeholder with a stable, known id that race-losers' `pollForFilledGreeting` can find. A parallel client-side row with the AI-generated id is written by `autoSaveMessages`; `loadMessages` dedup handles display.
- **consumeStream() added alongside skip:** `result.consumeStream()` is required for `onFinish` to fire when the response stream is not fully consumed client-side (e.g., client disconnects). Without it, placeholder remains empty.
- **Role-flip remount not eliminated by composite key:** Per research, the `useSelf` hydration causing `isFacilitator` to flip swaps between `ParticipantChatPanel` and `ChatPanel` (different component types). The composite key prevents cross-step `useChat` state reuse but does not prevent this role-flip remount — by design. The DB-lock greeting (defense 1), scope assertion (defense 2), and `stop()` cleanup (defense 3) are the actual safety net.

## Deviations from Plan

### Orchestrator-Applied Regression Fix

**1. [Rule 1 - Bug] Greeting placeholder never filled — two interacting issues**

- **Found during:** Task 7 human-verify checkpoint (mid-verification)
- **Issue 1:** `fillGreetingPlaceholder` in `onFinish` was guarded by `&& assistantMessageId`. Because AI SDK v5 generates the message id client-side, `assistantMessageId === ''` on the server, so the guard always failed and the placeholder was never filled.
- **Issue 2:** `abortSignal: req.signal` was applied to all requests including `__step_start__`. React Strict Mode (dev) double-invokes effects, causing an unmount immediately after mount, which aborted the in-flight stream and prevented `onFinish` from firing — leaving the placeholder empty.
- **Fix:** (a) Dropped `assistantMessageId` guard; pass `greetingClaim.messageId` (placeholder's deterministic id) to `fillGreetingPlaceholder`. (b) Skip `abortSignal` for `isStepStartTrigger` requests; preserve it for user-typed messages. (c) Added `result.consumeStream()` to guarantee `onFinish` fires on client disconnect. (d) Manually deleted 1 stale empty placeholder row in dev DB.
- **Files modified:** `src/app/api/chat/route.ts`
- **Committed in:** `2489368` (orchestrator-applied fix, committed by orchestrator mid-verification)

## Known Limitations & Follow-Ups

### 1. Inherited: response_message_id stays null (AI SDK v5, from Plan A)

> **Status: CLOSED in phase 62.2 (2026-05-16)** — Resolved alongside Plan A's documented limitation. AI SDK v6's `generateMessageId` is now passed to `toUIMessageStreamResponse`, populating `response_message_id` at insert time. See `62.1-01-SUMMARY.md` for the closure note and `.planning/phases/62.2-ai-sdk-v5-message-id-resolution/62.2-01-SUMMARY.md` for full details. Text below preserved as historical context.

The won-greeting fresh-gen path inherits Plan A's documented limitation. In `onFinish` server-side, `assistantMsg.id === ''` because AI SDK v5 generates the message id client-side. The `response_message_id` backfill UPDATE therefore sets `null`. This is benign for the admin UI (joins by scope+timestamp). The race-loser replay path uses `filled.messageId` directly and is unaffected.

Recommended fix paths (see 62.1-01-SUMMARY.md "Known Limitations" for full detail):
1. Route backfill through `autoSaveMessages` (client-side, knows the AI-generated id)
2. Investigate `streamText`'s `messageId` override option in AI SDK v5

### 2. req.signal abort effectiveness for user-typed messages not measured

`abortSignal: req.signal` is wired for user-typed messages (non-`__step_start__`) and `consumeStream()` is present. However, the actual reduction in `output_tokens` for an aborted user-typed request was not measured during verification. If token savings are important, a targeted measurement (start streaming, close tab, query `ai_usage_events.output_tokens`) is recommended as a follow-up.

### 3. Test runner gap — vitest not installed

Integration tests in `src/app/api/chat/__tests__/` and `src/lib/ai/__tests__/` require `npm install -D vitest` before they can run. Tests are written and correct — they are executable specifications pending the install.

### 4. Dual-row state from greeting fill

> **Status: CLOSED in phase 62.2 (2026-05-16)** — Resolved by `fillGreetingPlaceholder` now using the real AI SDK-generated id (server-side `generateMessageId`) instead of the deterministic `greeting:…` placeholder. The won-greeting path's UPDATE replaces the placeholder's messageId with the AI SDK id; the client's `useAutoSave` subsequently writes a row with the SAME id, which the unique index `chat_messages_session_step_participant_message_uniq` then drops via `onConflictDoNothing()`. Exactly 1 assistant row per scope post-greeting — verified by `scripts/verify-message-id-resolution.ts` AC-4. Text below preserved as historical context.

The won-greeting path fills the placeholder row with the deterministic id (`greeting:...:fac`) and the client's `autoSaveMessages` writes a parallel row with the AI-generated id. This dual-row state is benign: `pollForFilledGreeting` queries by `content > 0` (not by specific id), and `loadMessages` dedup handles UI display. A future cleanup could deduplicate these rows, but it is not required for correctness.

### 5. Role-flip remount not eliminated

Per research, the `useSelf` hydration causing `isFacilitator` to flip swaps between `ParticipantChatPanel` and `ChatPanel`. The composite key does not prevent this (different component types remount regardless of key). Defenses 1+2+3 (DB lock + scope assertion + stop()) prevent duplicates despite the remount. The composite key is defense-in-depth for cross-step state reuse only.
