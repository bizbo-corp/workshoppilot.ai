## Summary

The phase has a complete diagnosis already (CONTEXT.md), so this research is narrowly scoped to four planning prerequisites. Findings:

1. **Drizzle migrations** in this repo are 100% raw SQL, generated via `drizzle-kit generate`, applied manually (no auto-apply on deploy). Pattern is mature: 22 migrations, includes a precedent for dropping a unique index on `chat_messages` (migration 0018). No CHECK constraints exist yet — Postgres SQL syntax is the reference. **Canonical command:** `npm run db:generate` then commit the generated `.sql` file in `drizzle/`.

2. **AI SDK `useChat().stop()`** does abort the client-side fetch (calls `AbortController.abort()` internally on the in-flight request). However, when the `transport` memo recreates with new dependencies, the previous in-flight fetch is **NOT** automatically aborted — the old transport's pending request runs to completion and `onFinish` may fire, persisting the greeting. The fix requires explicitly calling `stop()` in a `useEffect` cleanup keyed on `sessionId`/`workshopId`. Workstream 3(b)'s `key={...}` approach forces a full unmount which IS a documented stop trigger ("closing a tab, refreshing the page, or calling the stop() function will trigger an abort signal"), but verify by testing — defense-in-depth requires both.

3. **Liveblocks remount hypothesis is CONFIRMED.** `MultiplayerRoomLoader` uses `next/dynamic({ ssr: false, loading: ... })`. The server SSRs the `loading` placeholder; on the client, the chunk loads, then the real `<MultiplayerRoom>` (with its `<RoomProvider>` tree) mounts. The `<StepContainer>` (and therefore `<ChatPanel>`) is passed as `children` of `<MultiplayerRoomLoader>`, which means the children mount inside the `loading` fallback's parent slot first, then **mount again** inside the real `<MultiplayerRoom>` tree once the chunk resolves. This is the dominant cause of the duplicate-greeting race in production. Workstream 3(b)'s `key` is necessary but **NOT sufficient** — the remount itself happens regardless of key. The greeting singleton (workstream 2) and stale-fetch abort (workstream 3a) are the primary defenses; the key just ensures intra-step navigation cleanly resets useChat state.

4. **REQUIREMENTS.md does NOT exist** at either `.planning/REQUIREMENTS.md` or repo root. Phase directory IS correctly created. STATE.md still shows `status: between_milestones, milestone: null` — needs updating to reflect 62.1 in_progress before kicking off implementation.

**Primary recommendation:** Plans should reference inline requirement IDs (DIAG-01, GREET-01, SCOPE-01, HALL-01, HIST-01) directly in their PLAN.md files. Do NOT create a REQUIREMENTS.md for a hotfix patch — it would pollute the next-milestone planning surface. The five workstreams are large enough to warrant 4–5 separate PLAN.md files (workstreams 1, 2, 3, 5 each as one plan; workstream 4 either bundled with 3 or split into a follow-on 62.2 per Claude's discretion).

### Pattern: Add a CHECK constraint (workstream 2: `length(message_id) > 0`)
**No precedent in repo** — there are zero CHECK constraints across all 22 migrations. Drizzle ORM 0.45 supports CHECK via the `check()` table modifier, but it generates idempotency-unsafe DDL. **Recommendation: hand-author the CHECK as a manual addition to the generated migration**, gated by a one-shot cleanup of existing offenders. Add to the same migration file as the cleanup so they're atomic.

Worked example (append to the migration generated for the schema-side `check()`):
```sql
-- Idempotent cleanup: remove any chat_messages rows with empty message_id
-- (defensive: there are 4+ known offenders from the duplicate-greeting bug)
DELETE FROM "chat_messages" WHERE "message_id" = '';
--> statement-breakpoint
-- Add CHECK constraint — IF NOT EXISTS guard makes the migration safe to re-run
ALTER TABLE "chat_messages"
  DROP CONSTRAINT IF EXISTS "chat_messages_message_id_nonempty_chk";
--> statement-breakpoint
ALTER TABLE "chat_messages"
  ADD CONSTRAINT "chat_messages_message_id_nonempty_chk" CHECK (length("message_id") > 0);
```

The `DROP CONSTRAINT IF EXISTS` + `ADD CONSTRAINT` pair is the standard Postgres idempotency pattern. (Postgres has no `ADD CONSTRAINT IF NOT EXISTS`; the drop-then-add is the canonical workaround.)

In the Drizzle schema file, also add (so future migrations stay aligned):
```typescript
import { check, sql } from 'drizzle-orm';
// ...inside the (table) => ({...}) block:
messageIdNonempty: check('chat_messages_message_id_nonempty_chk', sql`length(${table.messageId}) > 0`),
```

### Pattern: Drop a unique constraint (workstream 4: `step_summaries.unique(workshop_step_id)`)
Direct precedent: 0018 dropped a unique INDEX on chat_messages. The unique on step_summaries was created via `unique()` in the schema (table-level CONSTRAINT, named `step_summaries_workshop_step_id_unique`). Drop SQL:

```sql
-- Drop the unique constraint that prevents append-only history
ALTER TABLE "step_summaries" DROP CONSTRAINT IF EXISTS "step_summaries_workshop_step_id_unique";
--> statement-breakpoint
-- Add the new columns
ALTER TABLE "step_summaries" ADD COLUMN "is_current" boolean DEFAULT true NOT NULL;
--> statement-breakpoint
ALTER TABLE "step_summaries" ADD COLUMN "superseded_at" timestamp (3);
--> statement-breakpoint
-- Backfill: any existing rows are by definition the current version
UPDATE "step_summaries" SET "is_current" = true WHERE "is_current" IS NULL;
--> statement-breakpoint
-- Partial unique index — only one is_current row per workshop_step
CREATE UNIQUE INDEX "step_summaries_current_workshop_step_uniq"
  ON "step_summaries" USING btree ("workshop_step_id") WHERE "is_current" = true;
```

The partial unique index (`WHERE is_current = true`) is critical: it preserves the "exactly one active summary per step" guarantee while allowing unbounded historical rows. Drizzle 0.45 supports partial indexes via `.where()` on `uniqueIndex()` — check the schema file for stepSummaries and add:
```typescript
currentUniq: uniqueIndex('step_summaries_current_workshop_step_uniq')
  .on(table.workshopStepId)
  .where(sql`${table.isCurrent} = true`),
```

### Pattern: Add nullable columns (every migration)
Trivial. ALTER TABLE ADD COLUMN with no `NOT NULL` is the default. See 0019, 0021. No backfill needed.

### Pattern: Idempotent data cleanup
Use `IF EXISTS` / `IF NOT EXISTS` guards, set-based UPDATEs with `WHERE ... IS NOT NULL` predicates, and `ON CONFLICT DO NOTHING` for inserts. 0022's backfill `UPDATE "workshops" SET "tier" = 'solo', "tier_paid_at" = "credit_consumed_at" WHERE "credit_consumed_at" IS NOT NULL AND "tier" IS NULL;` is the template — `WHERE col IS NULL` makes it a no-op on re-run.

For workstream 1's TTL cleanup (14 days), recommend a separate scheduled job rather than inline migration logic. A simple Vercel Cron route + `DELETE FROM chat_request_logs WHERE created_at < now() - interval '14 days'` is sufficient. **Confidence: MEDIUM** — repo doesn't appear to have existing cron patterns; planner should grep for `vercel.json` and `app/api/cron/*` to confirm.

### What `useChat().stop()` actually does — verified

| Question | Answer | Source |
|----------|--------|--------|
| Does `useChat()` return `stop()`? | **YES** — signature `stop: () => void`, no parameters | [AI SDK docs: useChat reference](https://ai-sdk.dev/docs/reference/ai-sdk-ui/use-chat) (HIGH) |
| What does `stop()` abort? | **Client-side fetch only** — calls `AbortController.abort()` on the in-flight HTTP request, closing the connection | [AI SDK chatbot docs](https://ai-sdk.dev/docs/ai-sdk-ui/chatbot) and [stopping streams](https://ai-sdk.dev/docs/advanced/stopping-streams) (HIGH) |
| Does it stop server-side generation? | **NO, not directly.** Server only stops if it observes the closed connection via `req.signal.aborted` and forwards to `streamText({ abortSignal })`. | [Troubleshooting: abort breaks resumable streams](https://ai-sdk.dev/docs/troubleshooting/abort-breaks-resumable-streams) (HIGH); confirmed by [Issue #9707](https://github.com/vercel/ai/issues/9707) (MEDIUM — the bug only affects createUIMessageStream's outer wrapper, not streamText) |
| Does `DefaultChatTransport` accept an `AbortSignal` in its constructor? | **Not directly via constructor body.** Transport accepts `headers`, `body`, `credentials`, `fetch` (custom fetch) per docs. AbortController is managed internally by `useChat`'s `stop()` hook. | [AI SDK transport docs](https://ai-sdk.dev/docs/ai-sdk-ui/transport) (MEDIUM — docs are incomplete; verified by absence in API reference) |
| When `transport` memo recreates, is the previous in-flight fetch auto-aborted? | **NO — confirmed by docs gap and behavioral evidence.** AI SDK docs explicitly do not document automatic cleanup on transport replacement. The previous transport's `sendMessages()` returns a promise that resolves on stream completion regardless. The new transport just becomes the target for *future* `sendMessage` calls. | LOW-MEDIUM — inferred from absence in [transport docs](https://ai-sdk.dev/docs/ai-sdk-ui/transport) and [migration-5 guide](https://ai-sdk.dev/docs/migration-guides/migration-guide-5-0); planner SHOULD validate by adding a console log to onFinish + a test that swaps body mid-stream. |
| What happens to `onFinish` if the stream is aborted? | When the client aborts via `stop()`, the stream errors out and `onError` fires (NOT `onFinish`). However, the server-side `onFinish` *may* still fire if the server completes generation before observing the abort. **Critical for persistence:** the existing `useAutoSave` debounced writer will persist whatever messages are in `messages` state at the time, regardless of abort. | [Stopping streams docs](https://ai-sdk.dev/docs/advanced/stopping-streams) (MEDIUM); behavioral verification needed |
| When component unmounts, does in-flight fetch abort? | **NO automatic cleanup.** `useChat` does not ship an unmount-effect that calls `stop()`. The fetch promise is held by the closed-over transport reference until it settles. | LOW — inferred; confirm by reading `@ai-sdk/react` source if available. |

### The exact code pattern to wire into `chat-panel.tsx` (and `participant-chat-panel.tsx`)

The fix has two complementary parts. Both are needed — neither is sufficient alone.

#### Part A — Call `stop()` on cleanup keyed by scope

Currently `chat-panel.tsx:930` destructures `{ messages, sendMessage, status, setMessages }` from `useChat`. Add `stop`:

```typescript
const { messages, sendMessage, status, setMessages, stop } = useChat({
  transport,
  messages: initialMessages,
  onError: (error) => { /* unchanged */ },
});

// NEW — abort any in-flight request when the chat scope changes.
// Without this, the previous transport's POST runs to completion, onFinish
// fires server-side, and the row writes against the OLD scope into the
// NEW scope's chat_messages (the cross-workshop leak vector).
React.useEffect(() => {
  return () => {
    // Cleanup runs when sessionId/stepId/workshopId change OR component unmounts.
    // stop() calls AbortController.abort() on the underlying fetch.
    stop();
  };
}, [sessionId, step.id, workshopId, stop]);
```

The closure over `stop` is safe because `useChat` returns a stable identity for `stop` (it's bound to the hook's internal AbortController). Including it in deps is React-best-practice and triggers no extra runs because the identity is stable.

#### Part B — Use `key` on the consumer to force unmount

Already specified in workstream 3(b). Even with stop() wired, mid-stream message state (`messages`, `status`) survives if React reuses the component instance across scope changes. The `key` forces a fresh mount with a fresh `useChat` state.

```tsx
// In step-container.tsx:1342 and :1353 — replace key={resetKey} with:
<ChatPanel
  key={`${sessionId}:${workshopId}:${step.id}:${resetKey}`}
  // ...rest unchanged
/>
```

Combining `${resetKey}` preserves the existing reset-button behavior (incrementing resetKey forces remount).

#### Part C — Server-side AbortSignal forwarding (defense-in-depth)

In `src/app/api/chat/route.ts`, forward the request's abort signal to Gemini so server-side generation actually stops when the client closes the connection. Currently `streamTextWithRetry` is called without an abortSignal. Add:

```typescript
export async function POST(req: Request) {
  // ... existing validation, scope assertion (workstream 3c), greeting singleton (workstream 2) ...

  const result = await streamTextWithRetry({
    model: chatModel,
    system: systemPrompt,
    messages: convertToModelMessages(messages),
    abortSignal: req.signal,  // ← NEW: cancels Gemini call when client disconnects
    // ... rest unchanged
  });
}
```

Per [Vercel changelog](https://vercel.com/changelog/node-js-vercel-functions-now-support-request-cancellation), Node.js Vercel Functions support `req.signal.aborted` since 2024. Combined with `streamText`'s native `abortSignal` option, this gives end-to-end cancellation.

**Confidence calibration:** stop() aborts the client fetch — HIGH. Transport-recreate doesn't auto-abort — MEDIUM (inferred from docs gap; should be empirically validated in implementation). Stop+key+server-signal pattern — HIGH (combination of documented mechanisms).

### Why CONTEXT.md's claim "use AbortController in the transport memo" is slightly misleading

The CONTEXT.md decision says "Abort in-flight `useChat` requests when `sessionId` or `workshopId` changes — use `AbortController` in the `transport` memo and call `useChat`'s `stop()` on cleanup." The transport memo itself doesn't expose an AbortController slot — `DefaultChatTransport` doesn't accept one in its constructor. The actual mechanism is **the `stop()` cleanup pattern shown in Part A above**, which uses `useChat`'s INTERNAL AbortController. The transport memo just needs to recreate (which it already does — see chat-panel.tsx:927) so future requests target the new scope. The planner should reframe the plan as "wire `stop()` into a useEffect cleanup" rather than "instantiate a new AbortController in the transport memo."

## Liveblocks Remount Investigation

### Hypothesis: CONFIRMED

`<ChatPanel>` (and `<ParticipantChatPanel>`) mount **at least twice** in production multiplayer workshops. Evidence:

### The mount path
From `src/app/workshop/[sessionId]/step/[stepId]/page.tsx:1208-1244`:
```tsx
{session.workshop.workshopType === 'multiplayer' ? (
  <MultiplayerRoomLoader workshopId={...} sessionId={sessionId}>
    <StepContainer ... />     ← children prop, contains <ChatPanel>
  </MultiplayerRoomLoader>
) : (
  <StepContainer ... />        ← solo: direct mount
)}
```

From `src/components/workshop/multiplayer-room-loader.tsx:17-27`:
```tsx
const MultiplayerRoom = dynamic(
  () => import('@/components/workshop/multiplayer-room'),
  {
    ssr: false,                              // ← critical
    loading: () => <div>Connecting to live session...</div>,
  }
);

export function MultiplayerRoomLoader({ workshopId, sessionId, children }) {
  return <MultiplayerRoom workshopId={workshopId} sessionId={sessionId}>{children}</MultiplayerRoom>;
}
```

### Lifecycle (verified against [Next.js lazy-loading docs](https://nextjs.org/docs/app/guides/lazy-loading))
`next/dynamic` is "a composite of React.lazy() and Suspense" (per Next.js docs). With `ssr: false`:

1. **Server render:** `MultiplayerRoomLoader` returns null/fallback. Children passed to `MultiplayerRoom` are NOT rendered server-side because the server skips the dynamic component entirely. Result: server HTML has only the loading shell.
2. **Client hydration:** React hydrates the loading shell. The `loading: () => <div>Connecting...</div>` fallback renders.
3. **Chunk loaded:** The dynamic import resolves. React swaps the loading fallback for the real `<MultiplayerRoom>` tree, which now renders `<RoomProvider>` wrapping the `children` prop (`<StepContainer>` containing `<ChatPanel>`).
4. **Result:** `<ChatPanel>` mounts for the first time HERE (it never rendered during step 1 or 2 because the children were under the dynamic boundary, which renders the fallback, not the children).

So in multiplayer the ChatPanel mounts **once** post-hydration — but a second mount cycle can occur because:

### The second mount: `useSelf()` re-render inside `MultiplayerRoomInner`
From `src/components/workshop/multiplayer-room.tsx:298-330`:
```tsx
function MultiplayerRoomInner({ children, sessionId, workshopId }) {
  const self = useSelf();   // ← null until WebSocket connects
  return (
    <MultiplayerContext.Provider
      value={{
        participantColor: self?.info?.color ?? null,
        isMultiplayer: true,
        isFacilitator: self?.info?.role === "owner",
        // ...
      }}
    >
      {/* renderless listeners + */}
      {children}
    </MultiplayerContext.Provider>
  );
}
```

`useSelf()` returns `null` until the Liveblocks WebSocket handshake completes. Then it returns the participant object. **This causes the Provider's `value` to change**, which propagates to consumers but **does not by itself cause a remount of `children`**. Children re-render only if they consume the context. `<ChatPanel>` doesn't appear to consume `MultiplayerContext` directly (verified via grep — only `useMultiplayerContext` callers are inside the room component's own listeners).

So the `useSelf` hydration doesn't remount ChatPanel. **But there's still a real remount path:**

### The actual remount: facilitator-vs-participant branching in `step-container.tsx:1340-1357`
```tsx
{isMultiplayer && !isFacilitator && effectiveParticipantId ? (
  <ParticipantChatPanel key={resetKey} ... />
) : (
  <ChatPanel key={resetKey} ... />
)}
```

`isFacilitator` is computed from `useMultiplayerContext()` (see step-container.tsx, search for `isFacilitator`). On initial mount inside `MultiplayerRoomInner`, `useSelf()` is null → `isFacilitator: false` (from the context default). After WebSocket handshake → `isFacilitator: true` (for the workshop owner). **This branch flip swaps `<ChatPanel>` ↔ `<ParticipantChatPanel>` — a different component type, which is a hard remount with full unmount-then-mount lifecycle.**

For a facilitator opening their own workshop:
- t=0ms: `MultiplayerRoomLoader` renders loading. ChatPanel NOT yet mounted.
- t≈100ms: chunk loads. `MultiplayerRoomInner` mounts. `useSelf()` returns null. `isFacilitator: false`. **`<ParticipantChatPanel>` mounts.** Auto-start fires (`hasAutoStarted.current = true`). `__step_start__` POST sent. `/api/chat` begins streaming.
- t≈400ms: Liveblocks WebSocket settles. `useSelf()` returns owner. `isFacilitator: true`. **`<ParticipantChatPanel>` UNMOUNTS, `<ChatPanel>` MOUNTS.** Fresh `useChat` state, fresh `hasAutoStarted` ref. Auto-start fires AGAIN. Second `__step_start__` POST sent.

The first POST's `onFinish` writes the assistant greeting; the second POST's `onFinish` writes another greeting. **This is the duplicate-greeting race.** The cross-workshop leak compounds when the user has multiple tabs — the in-flight first POST may have a stale body if the user navigated, and the second POST has fresh body.

### Defense-in-depth status

| Workstream | Defense | Sufficient against this remount? |
|------------|---------|----------------------------------|
| 2 (DB-lock greeting singleton) | `INSERT ... ON CONFLICT DO NOTHING` on placeholder row | **YES — primary defense.** Even if both POSTs reach the API, the second loses the race and replays. |
| 3(a) (`stop()` on cleanup) | Aborts the first POST when ParticipantChatPanel unmounts | **YES — kills the symptom at the source.** The first POST is aborted before it can persist. |
| 3(b) (`key` on ChatPanel) | Forces remount on scope change | **NO** — doesn't address the type-flip remount. The key is for a different bug (cross-step navigation reusing useChat state). |
| 3(c) (server-side scope assertion) | Returns 409 if `sessionId.workshopId !== body.workshopId` | **YES — catches cross-tab leaks specifically.** |

### Recommendation: Don't try to eliminate the remount upstream

The `useSelf()` hydration is fundamental to Liveblocks; we can't avoid it. Two alternative architectures were considered and rejected:

1. **Render a static loading state until `useSelf()` resolves** — would harm perceived performance; the chat is the primary UX.
2. **Determine isFacilitator server-side and pass as a prop** — already done (the page passes participant state). But the *Liveblocks self* is the source of truth for runtime role; server-side Clerk check + Liveblocks self need to be reconciled.

**Better:** Stabilize the role determination by computing `isFacilitator` from the SERVER prop (Clerk userId === workshop.clerkUserId) as the primary source, and only use Liveblocks `self` for color/displayName. This eliminates the type-flip entirely. **However, this is out of scope for 62.1** per CONTEXT.md ("workshop/session schema changes... auth flow changes" out of scope). Workstreams 2 + 3(a) + 3(c) are the right defenses for this phase.

Same fix needed in both files. The `useEffect` cleanup pattern from Part A above applies identically to participant-chat-panel.tsx.
