## Known Limitations (Classified as Acceptable)

These items were identified during execution and explicitly documented in the SUMMARYs. They are **not gaps**.

### 1. `response_message_id` reliably null in fresh-gen rows (AI SDK v5)
AI SDK v5 generates the assistant message id client-side. The server's `onFinish` receives `assistantMsg.id === ''`. The backfill UPDATE skips on empty string, leaving `response_message_id = null`. Admin UI uses scope+timestamp join (not `response_message_id`), so the observability goal is fully met. Race-loser replay rows populate `response_message_id` directly from `filled.messageId` and are unaffected. Documented in `62.1-01-SUMMARY.md` and in-code at `route.ts` line ~254.

### 2. Test runner gap — vitest not installed
`package.json` has no unit test runner (only Playwright for e2e). The three test files in `src/app/api/chat/__tests__/` and `src/lib/ai/__tests__/` and `src/lib/context/__tests__/` are correctly authored vitest-compatible specifications. They require `npm install -D vitest` to run. The runnable `scripts/verify-sentinel.ts` works today via `npx tsx`. Documented in `62.1-02-SUMMARY.md`.

### 3. 5x-mount / cross-tab / req.signal abort not exhaustively re-run after regression fix
The regression fix (commit 2489368) addressed the two interacting root causes (empty-id guard + `abortSignal` on greeting). The manual scenarios from Task 7 were accepted as deferred rather than re-run at the checkpoint. The mechanisms are wired; the scenarios need one final human confirmation.

### 4. TTL cleanup for `chat_request_logs` deferred
No `vercel.json` cron or `src/app/api/cron/` infra in repo. Data accumulation is benign at current usage rates. Flagged as follow-up in `62.1-01-SUMMARY.md`.

### 5. Dual-row state from greeting fill
The won-greeting path writes to the placeholder row with the deterministic id; the client's `autoSaveMessages` writes a parallel row with the AI-generated id. `loadMessages` dedup handles display. Not a correctness issue.

### 6. Role-flip remount not eliminated
Per research, `useSelf` hydration causes `isFacilitator` to flip, swapping component types. Composite key does not prevent this. DB-lock (defense 1) + scope assertion (defense 2) + `stop()` (defense 3) are the actual safety net. Correctly documented as expected behavior, not a regression.
