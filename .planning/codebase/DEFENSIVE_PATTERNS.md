# Defensive Patterns

Hard-won fixes that protect specific subsystems from regression. **Do not strip a defense without a regression test that proves the underlying failure no longer occurs.** Each entry below patches a real, reproduced incident.

This document is hand-maintained (not auto-generated by `/gsd:map-codebase`). Add to it whenever a fix exists primarily to prevent the same bug from coming back.

---

## Per-step AI greeting trigger (chat auto-start)

When a workshop step opens, the AI sends an opening message. The trigger flow spans the client (`useChat` from `@ai-sdk/react`), `/api/chat`, a DB-backed singleton (`message-persistence.ts`), and Gemini's stream. It has fallen back to a broken state several times. The system survives only because of the seven defensive layers below.

**Affected files**

- `src/components/workshop/chat-panel.tsx` (facilitator)
- `src/components/workshop/participant-chat-panel.tsx` (participant)
- `src/app/api/chat/route.ts`
- `src/lib/ai/message-persistence.ts`
- `src/lib/ai/prefetch-greeting.ts`
- `src/actions/auto-save-actions.ts` (`refetchStepMessages` server action)

### Failure modes this protects against

- "AI is thinking forever" — `useChat` status stuck in `submitted`/`streaming`, no chunks arrive.
- Duplicate greetings (one per Strict-Mode mount, one per tab, etc.).
- Stuck-permanent empty placeholder rows blocking the singleton.
- Cross-scope message leaks (a stream completing into the wrong session/step).
- Stream completing server-side but the client `useChat` never receiving the assistant message.
- Duplicate-detection and replay logging breaking due to missing message IDs.

### The seven defenses

1. **Scope-comparison effect (NOT cleanup-on-unmount) to call `stop()`.**
   `chat-panel.tsx:993-1006`, `participant-chat-panel.tsx:205-216`. React 19 Strict Mode dev mounts → cleans up → remounts; a cleanup-based `stop()` aborts the in-flight `/api/chat` fetch before the server sees it. Instead, track the previous `(sessionId, stepId, workshopId)` tuple in a ref and only call `stop()` when scope **actually** changes. Strict-Mode remounts keep the same scope, so no abort.

2. **Auto-refetch on empty-stream completion.**
   `chat-panel.tsx:1020-1042`, `participant-chat-panel.tsx:265-290`. When `useChat` status transitions submitted → ready with zero assistant content in client state, fetch persisted messages from DB via `refetchStepMessages` and `setMessages()` them in. 250ms debounce so any genuinely in-flight chunks have a chance to land first. Treats the AI SDK v6 symptom where the stream completes server-side correctly but the client never gets the message pushed.

3. **45s stuck-state recovery toast.**
   `chat-panel.tsx:1048-1080`, `participant-chat-panel.tsx:295-325`. If status stays in submitted/streaming with no chunks for 45 s, show a persistent Sonner toast with a Retry action. Retry stops the call, clears local messages, re-arms auto-start.

4. **`deleteStaleEmptyPlaceholders` sweep at top of `claimGreetingPlaceholder`.**
   `message-persistence.ts:145-184`. Backstop for orphaned empty placeholder rows older than 60 s (= `/api/chat` `maxDuration`). Without this, the DB-lock singleton can stay permanently held when prefetch or `onFinish` dies silently.

5. **Server-side scope assertion at `/api/chat`.**
   `route.ts:62-85`. Returns 404/409 on session/workshop/step mismatch. This is what makes it **safe** to relax the client-side abort guard in layer 1 — cross-scope leak protection lives server-side, not on the client.

6. **DB-lock greeting singleton.**
   `claimGreetingPlaceholder` / `pollForFilledGreeting` in `message-persistence.ts`. Only one client wins the claim; losers poll for the filled row. Prevents duplicate greetings under Strict-Mode double-mount, multiple tabs, multi-device, or repeat-fetch retries.

7. **Server-side message-ID resolution (phase 62.2).**
   `generateMessageId` wired through, `responseMessageId` hoisted and populated in `chat_request_logs` at insert, `saveMessages` does **not** filter empty-id rows (the CHECK constraint is authoritative). Verified by `scripts/verify-message-id-resolution.ts`. Without this, duplicate detection and replay logging break.

### Diagnostic logging

Structured `[greeting-lifecycle]` logs span prefetch, `/api/chat`, and both chat panels (claim/poll outcomes, Gemini durations, fresh-gen fall-through, `onFinish` content length, client status transitions). **Do not strip these.** They are the only way to triage the next regression. Per-render `console.log` spam was a separate concern and was removed in commit `cddbf45` — keep that distinction (lifecycle logs are scoped and intentional; render spam is not).

### Anti-patterns

| Idea | Why it's wrong |
|------|----------------|
| "Just `stop()` in cleanup, it's cleaner." | Re-introduces the Strict-Mode race in layer 1. |
| "Remove the 60 s stale-placeholder sweep." | Re-introduces stuck-singleton state. |
| "Filter empty-id rows in `saveMessages`." | Already tried (61.2-02). Broke message-ID resolution. The CHECK constraint is authoritative. |
| "Replace the DB lock with an in-memory flag / Redis without persistence." | Breaks across tabs, devices, and serverless cold starts. |
| "Strip the lifecycle logs, they're noisy." | They scope per request and are the only triage signal. Strip render-loop logs instead. |

### When you do need to touch this code

Land a regression test first. The closest reference is `scripts/verify-message-id-resolution.ts`. Reproduce the failure you're fixing in a test before relaxing any of the seven layers.
